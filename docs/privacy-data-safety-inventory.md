# Hisaab 2.0 Privacy & Data Safety Inventory

Prepared for future Android and Google Play Store readiness. This inventory is based on inspection of the current React/Vite codebase, Supabase data access layer, local storage usage, export/import utilities, and app settings screens.

## 2. Local Device Storage

### localStorage

Observed keys include:

- `hisaab_supabase_uid`: cached current Supabase user id for DB helper calls.
- `hisaab_user_name`: cached profile/display name.
- `hisaab_primary_currency`: cached primary currency.
- `hisaab_onboarded`: onboarding completion flag.
- `hisaab_data_version`: local app data version.
- `hisaab_pending_invite`: invite token captured before login.
- `hisaab_app_mode`: `splits_only` or `full_tracker`.
- `hisaab_lang`: UI language.
- `hisaab_pin_hash`: SHA-256 hash of local PIN plus a static app salt.
- `hisaab_identifier`: local PIN/account identifier.
- `hisaab_mobile`: mobile number stored from Settings profile UI.
- `hisaab_backfill_persons_v1`, `hisaab_backfill_persons_v1:lock`: migration flags.

Logout cleanup clears most user-scoped localStorage keys through `resetAllUserStores()`, but `hisaab_mobile` and `hisaab_lang` are not currently listed in that cleanup array. `hisaab_supabase_uid` is cleared by auth state handling.

### sessionStorage

- `hisaab_pwa_dismissed`: dismisses the PWA install prompt for the current browser session.

### IndexedDB / Dexie

The app defines a Dexie database named `HisaabDB` with tables for accounts, transactions, loans, EMI schedules, goals, activity logs, upcoming expenses, split groups, group expenses, group settlements, and persons. Current data access in inspected code primarily uses Supabase through `supabaseDb.ts`; Dexie schema remains present and should be treated as possible local/offline storage in privacy disclosures until confirmed unused in production.

### Browser/Supabase Auth Storage

Supabase JS manages session persistence in browser storage by default. This includes session data such as access/refresh tokens required to keep the user logged in. The app also mirrors the current user id into `localStorage.hisaab_supabase_uid`.

## 3. Cloud Storage / Backend

Hisaab stores core app data in Supabase:

- Auth and email identity are handled by Supabase Auth.
- User profile and app preferences are stored in `profiles`.
- Personal finance records are stored in `accounts`, `transactions`, `loans`, `emi_schedules`, `goals`, `upcoming_expenses`, and `activities`.
- Contacts are stored in `persons`.
- Shared group data is stored in `split_groups`, `group_members`, `group_expenses`, `group_settlements`, `group_invites`, `group_events`, and `notifications`.
- Linked request workflows are stored in `linked_transaction_requests` and `linked_settlement_requests`.

Cloud storage is required for account sync, shared groups, linked user workflows, notifications, invite acceptance, and account deletion/anonymization.

## 4. Data Shared Between Users

Hisaab has collaboration features. In shared contexts, data visible to other users can include:

- Group name, currency, emoji, and members.
- Member display names, roles, connected/invited status, and joined state.
- Group expenses: description, amount, payer, splits, category, date, notes.
- Group settlements: payer/receiver members, amount, date, note.
- Group events: summaries of adds, edits, deletes, joins, and settlements.
- Group notifications targeted to connected members.
- Invite acceptance state and invite metadata, depending on group visibility.
- Linked transaction/settlement requests between two users, including amount, currency, note, status, and mirrored ledger ids.

Private personal accounts, private transactions, private goals, private upcoming expenses, and private contacts are not intended to be visible to other users except where the user explicitly initiates linked/group workflows or exports/shares backup data.

## 5. Data Not Collected

Based on current code inspection, Hisaab does not appear to collect or request:

- GPS or precise location.
- Address book/native device contacts.
- Camera/photos.
- Microphone/audio.
- SMS or phone call logs.
- Health or fitness data.
- Advertising ID.
- Biometric data.
- Installed apps.
- Web browsing history.

The app uses clipboard write operations for invite links/user codes/group codes. It uses file input for JSON backup import and browser download APIs for backup export. It does not appear to upload arbitrary photos/files.

## 6. Security / Privacy Controls

- Authentication is via Supabase Auth email/password.
- Supabase Row Level Security is present in schema/migrations for user-owned tables and shared group visibility.
- Soft-delete account deletion is implemented through `soft_delete_current_user`, keeping financial/group records intact while anonymizing profile identity.
- Deleted users are blocked on login/session restore by checking `profiles.is_deleted`.
- Logout clears in-memory Zustand stores and most user-scoped localStorage keys.
- Local PIN is only a local app lock. It is stored as a SHA-256 hash with a static salt in localStorage, not as a native secure enclave/keystore secret.
- Backup export produces a JSON file containing sensitive financial and settings data. After export, the user controls where that file is stored/shared.
- Backup import deletes and recreates user-owned cloud records for several tables. This is high-impact and currently guarded by a browser confirmation.
- Service worker/PWA caching exists and should be reviewed before native packaging to avoid stale cached shells.

## 8. Privacy Policy Draft Points

Include:

- App name: Hisaab 2.0 / Hisaab.
- Developer/legal entity and privacy contact email.
- What account data is collected: email, name, user id, public code, preferences.
- What financial data is collected: accounts, balances, transactions, loans, EMI schedules, settlements, goals, upcoming expenses.
- What collaboration data is shared with other users in groups and linked requests.
- What local data is stored on device, including local PIN hash and cached settings.
- What cloud provider is used: Supabase for auth/database/realtime.
- How backup export/import works and that exported JSON files may contain sensitive financial data.
- Account deletion policy: soft-delete/anonymization, shared financial records retained for accounting integrity.
- Data retention policy for private records, shared group records, deleted profiles, and backups.
- Security measures: authentication, RLS, HTTPS/cloud provider controls, logout clearing.
- User rights/contact method for data access, correction, deletion, and support.
- Whether data is sold/shared for advertising: current code shows no ad SDK or advertising ID.
- International transfer/storage note if Supabase region is outside user country.
- Effective date and update process.

## 9. Gaps / Risks

| Risk | Severity | Recommended Fix |
|---|---:|---|
| Privacy policy does not exist in repo | High | Create public privacy policy page before Play Store submission |
| `hisaab_mobile` not cleared by logout cleanup | Medium | Add it to `USER_SCOPED_LOCALSTORAGE_KEYS` if mobile remains user-scoped |
| Local PIN hash uses static salt and localStorage | Medium | For Android, consider native secure storage/biometric lock later |
| Backup exports highly sensitive JSON | High | Add clearer warning before export and document user responsibility |
| Backup import deletes/recreates user-owned records | High | Keep strong confirmation and test with shared group data |
| Dexie schema exists but current runtime use is unclear | Medium | Confirm whether IndexedDB is active or legacy before final disclosures |
| Soft-deleted auth user may still exist | Medium | Privacy policy must explain soft deletion and retained shared records |
| Shared records retain accounting history | Medium | Explain that shared group/settlement records remain visible to group members |
| Service worker caches shell/assets | Low-Medium | Review before Android packaging and disclose only if caching user data ever happens |
| Server/provider logs not documented | Medium | Include Supabase/Vercel operational logging in privacy policy if applicable |

## 10. Final Recommendation

Hisaab is close to being ready for a Play Store privacy declaration, but it should not be submitted until these are resolved:

1. Publish a real privacy policy URL.
2. Decide and document retention for soft-deleted users and shared financial records.
3. Clarify whether Dexie/IndexedDB is still actively used or legacy.
4. Add `hisaab_mobile` to logout cleanup if it remains user-scoped.
5. Add user-facing warnings for backup export/import sensitivity.
6. Re-check final Android build for any native SDKs, diagnostics, permissions, or device identifiers before completing Play Console Data Safety.
